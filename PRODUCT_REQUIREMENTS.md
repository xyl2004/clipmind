# ClipMind Product Requirements

本文档记录 ClipMind 下一阶段的产品需求。这里描述的是目标形态，不代表当前代码已经全部实现。

## Product Shape

ClipMind 的目标形态是一个 **macOS 悬浮式 AI 钱包**。

用户在浏览器、聊天、区块浏览器、文档或任意 macOS 应用中选中一段 Web3 内容后，通过 `Control + Option + W` 唤醒悬浮窗。悬浮窗应当成为主要钱包入口：用户可以围绕选中内容直接提问，也可以用自然语言让 AI 生成钱包操作计划，例如购买某个代币或向某个地址转账，并在悬浮聊天窗中查看完整确认单、点击本机签名按钮完成交易。

主窗口仍然保留，但它的定位更偏向钱包状态、服务连接、交易历史、对话历史、完整链上证据查看和高级管理。主窗口不是普通交易的必经复核步骤。

## Current Product State

当前产品已经具备这些基础能力：

- 读取 macOS 当前选中文字，并打开悬浮上下文对话窗。
- 围绕选中内容进行 AI 问答。
- 对钱包地址、代币合约、交易哈希和项目名补充 Surf / EVM 数据。
- 在主窗口中创建、导入和解锁本地 EVM 钱包。
- 将私钥保存在 macOS Keychain。
- 通过 Uniswap 生成交易确认单。
- 在用户确认后用本地钱包签名并广播交易。

当前还没有完整实现的目标能力是：**用户在悬浮窗中用自然语言发起钱包交易，并在悬浮窗中完成确认、签名和广播的主要流程**。

## Target Workflow

核心流程：

```text
选中内容
-> 唤醒悬浮窗
-> AI 理解上下文
-> 用户自然语言下达钱包指令
-> AI 生成结构化交易计划
-> App 做参数、余额、Gas、报价和风险检查
-> 在悬浮聊天窗中显示用户可读的完整确认单
-> 用户在悬浮窗中明确确认
-> 用户点击悬浮窗内的本机签名按钮
-> 本地钱包签名并广播
-> 显示交易结果和哈希
```

AI 负责理解意图和整理交易计划，但不能直接签名、不能绕过确认、不能接触私钥。

普通交易应当可以在悬浮窗内完成。主窗口用于管理、历史、完整证据和复杂情况复核，而不是每次交易的必经步骤。

## Primary Use Cases

### 1. Selected Address Transfer

用户选中一个钱包地址，然后在悬浮窗输入：

```text
给这个地址转 5 USDC
```

ClipMind 应当识别：

- 选中内容是收款地址。
- 用户意图是转账。
- 转账资产是 USDC。
- 转账金额是 5。
- 链需要根据当前钱包链、用户明说的链或默认策略确定。

执行前必须显示确认单：

- 动作：转账
- 链
- 收款地址
- 资产
- 金额
- Gas 预估
- 钱包余额是否足够
- 用户确认方式
- 本机签名并发送按钮

### 2. Selected Token Purchase

用户选中一个代币合约地址，然后在悬浮窗输入：

```text
用 20U 买这个币
```

ClipMind 应当识别：

- 选中内容是目标代币。
- 用户意图是购买代币。
- 支付资产通常是 USDC，除非用户明确指定其他资产。
- 支付金额是 20 USDC。
- 交易路径使用 Uniswap。

执行前必须显示确认单：

- 动作：购买代币
- 链
- 支付资产和金额
- 目标代币合约
- 预计收到数量
- 滑点
- Gas 预估
- 是否需要授权
- 报价有效期
- 风险检查结果
- 用户确认方式
- 本机签名并交易按钮

### 3. Plain Context Q&A

用户选中任意 Web3 文本，然后提问：

```text
这是什么意思？
这个地址有什么风险？
这个项目在做什么？
```

ClipMind 应当保持轻量问答体验，不强行进入交易流程。只有当用户明确表达购买、转账、兑换、授权等操作意图时，才进入交易计划流程。

### 4. Ambiguous Or Unsafe Intent

当用户指令不完整时，ClipMind 应当先追问，而不是猜测执行。

例子：

- 用户说“买一点这个币”，但没有金额。
- 用户说“转给这个地址”，但没有资产或金额。
- 用户选中的内容不是有效 EVM 地址。
- 当前链和目标代币链不明确。
- 钱包没有足够余额或 Gas。
- 报价过期。

这些情况都必须阻止签名，并要求用户补充或重新生成确认单。

## Floating Panel Requirements

悬浮窗是下一阶段的核心钱包入口，但仍然要保持简洁。

悬浮窗应当采用分阶段体验：

1. **Context State**
   - 显示选中内容的短预览。
   - 显示输入框。
   - 允许用户直接提问或下达钱包指令。

2. **AI Answer State**
   - 显示 AI 对选中内容的回答。
   - 如果有 Surf / EVM 摘要，只显示关键摘要，不展示完整证据表。

3. **Intent Draft State**
   - 当 AI 识别出交易意图后，显示结构化交易计划。
   - 用户能看懂这次操作准备做什么。
   - 缺少参数时先追问。

4. **Confirmation State**
   - 显示轻量但完整的确认单。
   - 必须明确展示动作、链、资产、金额、目标地址或代币、费用、风险和报价有效期。
   - 必须要求用户显式确认。
   - 确认单必须出现在聊天流中，而不是跳转到主窗口后才出现。
   - 确认单内必须提供清晰的本机签名按钮，例如“本机签名并发送”或“本机签名并交易”。

5. **Signing State**
   - 用户点击本机签名按钮后，悬浮窗显示签名中、广播中、等待结果等状态。
   - 签名按钮必须防止重复点击。
   - 如果报价过期、余额不足、Gas 不足、链不匹配或确认码错误，应阻止签名并在悬浮窗内显示原因。

6. **Execution Result State**
   - 显示交易是否已广播。
   - 显示交易哈希和区块浏览器链接。
   - 保存到交易历史。

悬浮窗不应该变成复杂交易所界面。它应当用聊天消息和操作卡片承载交易确认：内容完整，但布局紧凑。完整 Surf 证据、原始 JSON、服务配置、长历史列表和高级管理仍然放在主窗口。

## Main Window Requirements

主窗口的定位是钱包状态台和完整工作台。

左侧栏应参考轻量钱包控制台形态，而不是传统导航列表。它应当包含：

- App 标识：ClipMind。
- 本地钱包卡片：
  - 钱包名称，例如 Default Wallet。
  - 地址缩写和复制入口。
  - 本机签名状态标签。
  - 资产总额。
  - 连接状态，例如“已连接”。
- 服务连接：
  - Surf API。
  - OpenAI / B.AI。
  - Uniswap。
  - 每项右侧显示“已连接”“未配置”或错误状态。
  - 管理入口使用小按钮，不在左栏常驻展开 API Key 输入框。
- 交易历史：
  - 最近交易列表。
  - 类型图标，例如 Swap、Transfer、Approve。
  - 链和交易摘要。
  - 状态，例如成功、失败、等待中。
  - 时间。
  - “全部”筛选入口。
- 对话历史：
  - 当前会话高亮。
  - 最近上下文标题。
  - 时间。
  - 清空历史入口。

主内容区应当包含：

- 当前上下文和聊天详情。
- 完整 AI 中文解读。
- 完整 Surf / EVM 证据展示。
- 悬浮窗已完成交易的详情回放。
- 钱包管理、高级设置和服务配置。
- 调试信息和原始 JSON。

当悬浮窗中的交易意图比较复杂、风险较高或用户需要更多证据时，应当提供入口让用户打开主窗口查看完整详情。但普通转账和买币流程应当可以在悬浮窗内完成。

## AI Requirements

AI 层应当输出结构化交易意图，而不是直接执行动作。

结构化结果至少应包含：

- `action`: `ask`, `transfer`, `swap`, `unsupported`
- `selected_context`: 用户选中的原始内容
- `target_address`: 收款地址或目标合约地址
- `chain`: 链
- `spend_asset`: 支付资产
- `spend_amount`: 支付金额
- `recipient`: 收款地址
- `slippage`: 滑点
- `missing_fields`: 缺失参数
- `risk_notes`: 风险提示
- `requires_confirmation`: 是否需要确认
- `confirmation_summary`: 面向用户展示的确认单摘要

AI 不允许：

- 读取或输出私钥；私钥导出只能由用户在本地管理入口显式确认触发。
- 直接调用签名。
- 在用户确认前广播交易。
- 在确认单生成后悄悄修改关键参数。
- 对不明确的交易指令自行猜测金额、资产或链。

## Wallet And Execution Requirements

执行层必须由 App 和本地钱包控制，而不是由 AI 控制。

转账流程需要：

- 验证收款地址格式。
- 验证链和资产。
- 检查余额和 Gas。
- 生成待签名交易。
- 在悬浮窗中显示确认单。
- 用户在悬浮窗确认后本地签名广播。

买币流程需要：

- 验证目标代币合约地址。
- 检查支付资产和金额。
- 调用 Uniswap 报价。
- 检查授权需求。
- 检查报价新鲜度。
- 显示预计收到、Gas、滑点和风险提示。
- 在悬浮窗中显示确认单。
- 用户在悬浮窗确认后本地签名授权或兑换。

所有签名必须继续使用本地钱包和 macOS Keychain 中的私钥。

## Safety Requirements

任何链上执行都必须经过安全门槛：

- 用户必须看到确认单。
- 用户必须明确点击确认按钮。
- 用户必须点击悬浮窗内的本机签名按钮后才允许执行。
- 高风险操作必须输入地址后 4 位或同等确认码。
- 过期报价必须重新生成。
- 钱包余额不足时禁止签名。
- Gas 不足时禁止签名。
- 链不匹配时禁止签名。
- 地址无效时禁止签名。
- 缺少金额、资产或收款地址时禁止签名。
- LLM 不接触私钥。
- 私钥不写入日志、文档、源码或网络请求。
- 用户导出私钥必须二次确认，导出内容只短时显示在本地 UI 中，不进入 AI 上下文。

后续应增加策略检查：

- 单笔最大金额。
- 滑点上限。
- 可疑合约或黑名单地址提示。
- 授权额度提示。
- 代币合约风险提示。
- 新币、低流动性或异常交易行为提示。

## UX Principles

- 悬浮窗要快、轻、清楚。
- 悬浮窗可以完成普通交易，但必须通过聊天中的确认单和本机签名按钮完成。
- 钱包交易必须分阶段展示，不能像普通聊天一样直接执行。
- 用户说自然语言，系统展示结构化确认单。
- AI 的回答要短，交易确认要精确。
- 主窗口承接钱包状态、历史、设置和完整证据，悬浮窗承接高频问答和交易执行。
- 左侧栏应像钱包状态台，优先展示本地钱包、服务连接、交易历史和对话历史。
- 技术词保留英文，例如 EVM、Base、USDC、Gas、Uniswap、Surf。
- 关键操作中文表达清楚，例如“本机签名”、“确认转账”、“重新生成报价”。

## Acceptance Criteria

下一阶段实现完成时，应满足：

- 用户选中钱包地址后，可以在悬浮窗输入自然语言转账指令。
- 用户选中代币地址后，可以在悬浮窗输入自然语言购买指令。
- AI 能把自然语言转换为可检查的结构化交易计划。
- 参数不完整时，悬浮窗会追问补全。
- 任何交易执行前都会在悬浮聊天窗中出现完整确认单。
- 悬浮确认单包含本机签名按钮。
- 用户确认前不会签名或广播。
- 用户点击本机签名按钮后，可以在悬浮窗内完成普通转账或买币交易。
- 本地钱包继续通过 Keychain 管理私钥。
- 交易成功后在悬浮窗显示哈希，并写入交易历史。
- 用户可以从悬浮窗进入主窗口查看完整证据和历史。
- 主窗口左侧栏展示本地钱包卡片、服务连接状态、交易历史和对话历史。
- API Key、Sepolia 干跑、导出私钥、删除私钥等低频管理操作不常驻展开在左侧栏，而是进入管理或设置入口。

## Open Questions

- 默认链应该跟随用户当前选择、钱包最近使用链，还是根据 Surf 查询结果判断？
- 默认支付资产应优先使用 USDC，还是允许用户设置偏好资产？
- 高金额交易是否需要二次确认或强制打开主窗口复核？
- 单笔金额和滑点上限的默认策略是什么？
- 是否需要支持跨链转账或只做同链转账与同链 swap？
- AI 结构化意图应该由 B.AI 直接返回 JSON，还是先由 App 做规则解析再让 AI 补充解释？
